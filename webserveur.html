> TP - Objectif

Objectif : Faire un mini-blog

  • Fonctionnalités... fonctionnelles :
    • Gestion des articles en CRUD (Create, Read, Update, Delete)
      • Titre, Contenu, Auteur, date de création
    • Système de commentaires en CRUD
      • Contenu, auteur, date de création, article
    • Gestion (très basique) des utilisateurs
      • Nom d'utilisateur, mot de passe, isAdmin, isBanned
      • Merci de hasher le mot de passe en base !
    • Règles utilisateur :
      • Seul un administrateur peut créer/éditer/supprimer un article
      • Un membre banni ne peut pas créer de commentaire
      • Seuls les personnes connectées peuvent créer un commentaire
  • Fonctionnalités techniques :
    • Respect des verbes HTTP & des codes retour HTTP
    • Authentification (restez sur un truc simple, ne sortez pas une usine à gaz !)
    • Persistance de la donnée
    • Interface utilisateur "conviviale" (une GUI)
    • Session sans état (pas de session côté serveur)

Et si vous avez du temps :

  • Plusieurs tâches bonus, pour vous faire progresser, ajouter un peu de piment et monter vos notes de TP
  • Côté fonctionnel :
    • Gérer des réponses à des commentaires (voir les sites LinuxFR ou 20Minutes pour un aperçu détaillé du fonctionnement)
    • Demander à l'utilisateur de valider son mail quand il s'inscrit
    • Lister les articles par auteur
    • Ajouter un système de tag et de recherche par tag sur les articles
    • Système d'article épinglé
  • Côté technique :
    • Négociation de contenu (Par exemple, ajoutez la possibilité de ne pas récupérer une page HTML, mais sa représentation JSON ou XML)
    • Cache client et/ou serveur (si vous ne pouveez pas le faire pour raison techniques, expliquez la démarche)

> TP - Implémentation

tl;dr : vous êtes libres.

  • Vous choisissez votre/vos langage(s) :
    • Java, Golang, Node.js (meh), Python (meh²), PHP, Golo, Vert.x, ...
    • Evitez quand même les trucs trop ésotériques et/ou volontairement débiles (Brainfuck, Whitespace, Befunge, ...)
    • Pour le front : HTML/CSS classique, ou avec du JS, ou avec un Framework (Vue.js ou Angular par exemple)
  • Vous choisissez votre SGBD :
    • Relationnel : MariaDB, MySQL, Postgres, H2, SQLite, Oracle (non je rigole, interdiction d'utiliser Oracle), [...]
    • Document : MongoDB, RethinkDb, [...]
  • Essayez de NE PAS utiliser de Framework (style Spring, Laravel, etc) :
    • Un framework va abstraire l'implémentation de nombreuses parties du travail demandé
    • Vous ne progresserez pas si l'ordinateur fait tout le travail...
  • Architecture MVC (comme dans le cours) ou micro-services, vous décidez :
    • Après tout, si vous aimez vous faire du mal, qui suis-je pour vous en empêcher ?
    • Si vous partez sur du micro-services, vous avez le droit de tricher en ajoutant des headers pour désactiver le CORS
    • Je suis ouvert à toute autre idée d'architecture loufoque, tant que vous arrivez à la justifier (ou à l'expliquer)

Les conseils "du chef" :

  • Pour mon TP de Web-Serveur, j'avais utilisé :
    • Java (7 à l'époque, #JeSuisVieux)
    • Spark pour le serveur Web
    • FreeMarker pour les pages
    • Le package java.sql avec des PreparedStatement
    • H2 en configuration "stockage fichier"

> Ressources / Documents & infos utiles

  • Par où commencer ?
    1. Définir votre "stack" techno : quel base de données ? Quel architecture ? Quel code pour le serveur ? Si vous n'avez pas trop d'idée, le mieux serait de partir sur H2, FreeMarker et une architecture MVC. Les librairies sont disponibles sur Moodle.
    2. Puis, il vous faut réfléchir au modèle de données : Comment structurer l'information dans votre base de données ? Vous pouvez faire un diagramme UML pour les définir, par exemple.
    3. Ensuite, il vous faudra définir les adresses (endpoints) sur lesquelles les ressources seront disponibles : où créer un article ? où récupérer la liste d'articles ? etc.
    4. Une fois ces trois points réalisés, vous aurez terminé le "cadrage fonctionnel". L'heure sera donc venue de passer au code.
    5. En général, l'idée est de partir sur la strate la plus basse : définir vos objets, et les requêtes SQL pour accèder aux données.
    6. Ensuite, on va créer le code métier qui permet de récupérer un objet depuis l'extérieur, de vérifier ses critères, et de l'ajouter/modifier dans la base de données.
    7. Puis, on s'occupe des contrôleurs (la création des endpoints pour récupérer/créer/modifier les objets dans la base
    8. Enfin, on s'occupe de l'authentification.
